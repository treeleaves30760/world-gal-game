# Distributing to Mobile (Android APK + PWA, iOS PWA)

World Gal-Game's mobile story is built **entirely on top of the web build**
([distribution-web.md](distribution-web.md)). There is no separate native
mobile engine: a visual novel doesn't need native GPU access, so wrapping the
WASM/pygbag web bundle is both the pragmatic and the supported path.

This page is deliberately **honest about feasibility**. Some of what follows
is wired and unit-tested in this repo (the staging, the PWA generators);
some is wired but only verifiable on real hardware / a real pygbag toolchain
(the APK output, install behaviour on devices). Each section says which.

| Platform | Primary path | Secondary | App Store / native? |
|----------|--------------|-----------|---------------------|
| Android  | pygbag **APK** (WebView shell over the web bundle) | **PWA** ("Add to Home Screen") | Native buildozer SDL2 is **out of scope** |
| iOS      | **PWA only** (Safari "Add to Home Screen") | — | **No App Store path** |

> **Out of scope, on purpose:** native buildozer / python-for-android SDL2
> for Android, and any iOS App Store submission. There is no well-maintained
> pygame→iOS toolchain, and the App Store restricts interpreted code heavily.
> The NDK toolchain maintenance cost for buildozer is high enough that it is
> only a stretch goal if someone volunteers to own it.

---

## Shared foundation

Everything mobile reuses the web work already in the engine:

- **Async loop + responsive canvas** (`app.py`): the fixed 1280×720 logical
  canvas is letterboxed onto whatever viewport the device gives, and
  `run_async()` yields each frame so the tab/WebView never freezes.
- **Touch input** (`ui/input.py`): `FINGERDOWN` maps to tap → click/advance;
  a horizontal swipe sets `InputState.swipe` (left = advance, right = open the
  backlog). Coordinates are inverse-mapped from screen space to logical space
  so widget hit-testing is unchanged.
- **Persistent saves** (`platform_web.flush_storage()`): saves live under the
  IDBFS mount (`/data/<app_data_name>`) and are flushed to IndexedDB after
  every write. See the iOS caveat below — this is exactly where iOS bites.
- **PWA assets** (`web_pwa.py`): the Web App Manifest, a service worker, and
  the iOS `apple-mobile-web-app-*` meta tags. Generated by the web build and
  shared by both the Android install and the iOS install.

### Touch-friendly UI mode

`EngineConfig.touch_mode` (default **False**) is the flag intended to enable
finger-friendly hit targets on mobile. The recommended minimum tap-target
edge is `world_gal_game.config.MIN_TOUCH_TARGET_PX` (44 logical px, matching
Apple HIG / Google Material guidance).

Today the flag is a documented hook: desktop behaviour is byte-identical with
it off, and actual hit-target enlargement is a deliberate follow-up so this
milestone stays low-risk. When you wire enlargement, gate it strictly behind
`touch_mode` so mouse layouts never change.

## PWA assets (shared by Android + iOS web installs)

The web build emits, next to pygbag's `index.html`:

| File | Role |
|------|------|
| `manifest.webmanifest` | Web App Manifest — name, icons, `display: standalone`, orientation, `start_url`. Drives "Add to Home Screen" and a chrome-less launch. |
| `service-worker.js` | Offline app-shell cache. Caches the shell on install, serves cache-first with network fallback, evicts stale caches on activate. |
| `index.html` (patched) | The manifest `<link>`, the iOS `apple-mobile-web-app-*` `<meta>` tags + `apple-touch-icon`, and the service-worker registration `<script>` are injected into `<head>`. |

These are produced by pure generators in `world_gal_game/web_pwa.py`
(`pwa_manifest`, `service_worker_js`, `apple_meta_tags`, `inject_pwa_into_html`)
and written by `write_pwa_assets(...)` as a **post-build step** after pygbag
emits its output. The post-build step is resilient: if `index.html` isn't
found (a pygbag layout change, say) the manifest + worker are still written
and only the HTML patch is logged-and-skipped — it never fails an otherwise
good build.

### Icons

The manifest references `icon-192.png`, `icon-512.png`, and a maskable
`icon-512.png`; the iOS tags reference `apple-touch-icon` (`icon-192.png`).
The build does **not** synthesise binary PNGs (that would pull an image
dependency onto a path that must stay pure). Ship those icons in your output
directory — a missing icon only costs a generic launcher glyph, never a crash.
You can generate placeholders with the asset studio
(`world_gal_game.dev.asset_studio.placeholder_image`).

## Android — pygbag APK (primary)

```bash
uv sync --extra web        # pygbag (the [web] extra), same as the web build

# Build the APK (wraps the web bundle in a WebView shell):
uv run python build.py games/demo_pack --target android-apk
# or:  wgg build games/demo_pack --target android-apk
```

Output lands under `build/android/` (override with `--output`). The staging is
**identical** to a web build — same engine + pack tree + templated `main.py` —
then pygbag is invoked with `--build` plus its APK output flag
(`world_gal_game.build_web.PYGBAG_APK_FLAG`). The PWA assets are emitted into
the bundle as well, so the same artefact also supports "Add to Home Screen".

> **Honest note — unverifiable here.** This repo's environment cannot run
> pygbag or the Android toolchain, so the exact APK flag could not be executed
> to confirm. It is pinned as a single named constant with a comment so it
> lives in exactly one place; if a pygbag release renames it, fix
> `PYGBAG_APK_FLAG` and nowhere else. The **staging** that feeds it is fully
> unit-tested (`tests/test_build_android.py`) and is byte-identical to the web
> staging.

### Android — PWA (secondary, zero APK build)

You don't strictly need an APK on Android. Host the web build over HTTPS and
players can install it directly from Chrome:

1. Open the hosted URL in Android Chrome.
2. Menu → **Add to Home screen** (Chrome also shows an install banner when the
   manifest + service worker are present).
3. It launches standalone (no browser chrome), works offline (service-worker
   shell cache), and saves persist in IndexedDB.

## iOS — PWA only

There is **no App Store path** and **no native build**. The supported route is
Safari's "Add to Home Screen":

1. Open the hosted URL (HTTPS) in **Safari** (not an in-app browser).
2. Share sheet → **Add to Home Screen**.
3. It launches full-screen via the `apple-mobile-web-app-capable` meta tags;
   the status bar is `black-translucent` so the VN draws under the notch.

### iOS caveats — read before shipping

These are real iOS Safari limitations, stated plainly so players aren't
surprised:

- **IndexedDB eviction = saves can be lost.** iOS Safari evicts the IndexedDB
  of PWAs that go unused for ~7 days. Because saves live in IndexedDB (via the
  IDBFS mount), an infrequently-opened installed PWA **may lose its saves**.
  There is no engine-side fix; surface an in-game "export your save" affordance
  if save durability matters for your title, and tell players to relaunch
  periodically.
- **Audio is gated behind the first tap.** iOS blocks audio autoplay until a
  user gesture. The title-screen tap unlocks it — the same gesture the desktop
  and web builds already rely on — so this is usually invisible, but a title
  that tries to play BGM before any tap will be silent until the first touch.
- **No background execution / push.** A PWA is foreground-only; don't rely on
  timers running while backgrounded.

## Manual verification (on real devices)

The pure generators and the staging are unit-tested in CI. The **device**
behaviour can only be verified by hand:

1. **Build + serve over HTTPS.** Build the web bundle
   (`wgg build games/demo_pack --target web`) and host the output over HTTPS
   (a PWA requires a secure origin; `http://localhost` is treated as secure
   for desktop testing but a phone needs real HTTPS or a tunnel such as a
   dev TLS proxy).
2. **Install on the device.** Android Chrome → Add to Home screen; iOS Safari →
   Add to Home Screen. Confirm it launches standalone (no browser chrome) and
   shows your icon + name.
3. **Tap-advance.** Tap to advance dialogue; confirm a single tap advances one
   line.
4. **Swipe backlog.** Swipe right to open the backlog, swipe left to advance;
   confirm the gesture classification feels right.
5. **Save persistence across relaunch.** Make a save, fully close the PWA,
   relaunch from the home-screen icon, and confirm the save is still listed.
   On iOS, also note the eviction caveat above — a save surviving an immediate
   relaunch does **not** prove it survives a week of disuse.
6. **Offline launch.** With the service worker installed, enable airplane mode
   and relaunch; the app shell should still load.
